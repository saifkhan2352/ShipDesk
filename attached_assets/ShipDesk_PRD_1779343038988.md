## 1. EXECUTIVE SUMMARY

ShipDesk is an AI-native client portal built for freelance developers and small dev agencies that automatically generates plain-English weekly project status reports from GitHub activity, eliminating manual client communication overhead. It targets developers billing $3K–$15K/month who manage 2–10 simultaneous client projects and need a single, branded, professional interface to handle status reports, file sharing, invoicing, scope change requests, and client messaging. ShipDesk solves the core problem that developers lose an average of 7.4 hours/week on client admin — translating commits into updates, chasing invoices, and absorbing unbilled scope creep — by embedding directly into the GitHub workflow they already use.

---

## 2. PROBLEM STATEMENT

### 2.1 Current Pain Points

- **Manual status reporting:** Developers spend 2–4 hours/week reading through commit histories and translating them into client-readable English email updates, with no tooling to automate or accelerate this process.
- **Scattered communication:** Client conversations happen simultaneously across email, Slack, Notion, Google Docs, and WhatsApp, creating lost context, missed requests, duplicated responses, and conflicting version history.
- **Unpaid scope creep:** The average freelance developer loses $7,800–$15,600/year in unbilled work because scope changes are handled informally via email with no structured approval flow, quote step, or payment trigger.
- **No developer-native portal:** Existing client portals (Copilot, SuperOkay, HoneyBook) have zero integration with GitHub, Linear, or Vercel — the actual tools developers use daily — requiring double data entry and constant context-switching.
- **Unprofessional client presentation:** Developer output is high quality but client-facing communication (plain emails, Google Drive links, PayPal invoices) fails to reflect it, undermining the developer's perceived value and their ability to charge premium rates.

### 2.2 Proposed Solution

ShipDesk provides a single white-label client portal that connects to a developer's GitHub repositories via OAuth and webhooks, automatically ingesting commits, pull requests, and releases each week. Every Friday at 9:00 AM UTC — or on demand at any time — Google Gemini transforms that raw activity data into a polished, plain-English "This Week in Your Project" report delivered inside a branded portal the client accesses via a magic link with no password required. The same portal handles structured scope change requests with a built-in approval flow and Lemon Squeezy payment links, milestone invoicing with automatic webhook-confirmed status updates, drag-and-drop file sharing via Cloudinary, and a per-project messaging thread. The result is a professional, automated client communication layer that sits on top of the developer's existing workflow and eliminates the 7.4 hours/week of manual admin overhead.

## 5. TECH STACK & ARCHITECTURE

### 5.1 Recommended Stack

| Animations | Framer Motion | Report generation loading states, page transitions, onboarding checklist animations |
| Backend | Node.js + Express + TypeScript (`tsx` watch mode) | Familiar JS ecosystem, fast iteration, compatible with Prisma and all required third-party SDKs |
| Database | PostgreSQL + Prisma ORM | Relational integrity for project/client/invoice relationships; Prisma generates Railway-compatible Linux binaries |
| Auth | Clerk | Passwordless magic link for clients, email/Google OAuth for developers, `svix`-verified webhook events |
| File storage | Cloudinary | Managed upload widget, CDN delivery, signed upload parameters, no self-hosting required |

| Backend hosting | Railway | Managed PostgreSQL addon, persistent Node.js server, automatic `PORT` injection, straightforward Prisma migration support |

> **Security note:** `SESSION_SECRET` and `AES_ENCRYPTION_KEY` serve different cryptographic purposes and must be distinct values. `SESSION_SECRET` (≥ 32 chars, any random string) is used exclusively for JWT signing. `AES_ENCRYPTION_KEY` (exactly 64 hex chars = 32 bytes = 256 bits) is used exclusively for AES-256-CBC encryption of third-party OAuth tokens. Sharing a key across multiple algorithms is a cryptographic anti-pattern and must never be done. Generate with: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` — the output is exactly 64 hex characters.
>
> **Note on `CLOUDINARY_UPLOAD_PRESET`:** The preset must be configured in the Cloudinary dashboard as a **signed** preset (not unsigned). Server-generated signed upload parameters authorize each upload; the preset name is embedded in the signature request. Unsigned presets are never used.
>
> **Note on `LEMONSQUEEZY_PRODUCT_ID` / `LEMONSQUEEZY_VARIANT_ID`:** These values depend on the resolution of OQ-6. If Lemon Squeezy supports per-checkout price overrides on a single variant, one variant ID is sufficient. If not, a new product/variant must be created per checkout via the API and these env vars may not be used as-is. Confirm before beginning Feature 9 and 10 implementation.

### Feature 1: Developer Onboarding & Workspace Setup

- **Priority:** P0
- **User Story:** As a developer, I want to sign up, create my workspace with agency branding, and receive a unique client portal subdomain so that my clients see a professional, branded experience from day one.

### Feature 3: GitHub Repository Connection

- **Priority:** P0
- **User Story:** As a developer, I want to connect my GitHub account and link repositories to my projects so that ShipDesk automatically syncs commits, pull requests, and releases without any manual data entry.
- **Acceptance Criteria:**
  - [ ] Developer initiates GitHub OAuth from the project Settings tab; clicking "Connect GitHub" redirects to `GET /api/github/connect` which builds the GitHub OAuth URL with scopes `repo` and `read:user` and a CSRF `state` parameter stored in a short-lived (10-minute) signed cookie; developer is then redirected to GitHub
  - [ ] After GitHub authorization, the callback at `GET /api/github/callback?code=&state=` validates the `state` cookie, exchanges the `code` for an access token, encrypts it using AES-256-CBC with `AES_ENCRYPTION_KEY`, and stores the ciphertext in `GitHubConnection.accessTokenEncrypted`; developer is redirected to `/settings?github=connected`
  - [ ] After OAuth, developer can select a GitHub repository to link to a specific project; the repository list is fetched fresh from GitHub's `GET /user/repos` on every request (not cached in DB) and displayed in a searchable combobox showing: org/repo name, visibility badge (Public/Private), last pushed date
  - [ ] Linking a repository calls `POST /api/github/connect-repo`; this registers a webhook on the repository via GitHub's `POST /repos/{owner}/{repo}/hooks` listening for events `push`, `pull_request`, and `release`; the returned GitHub webhook ID is stored in `Project.githubWebhookId`
  - [ ] All incoming webhook payloads are validated via HMAC-SHA256 against `GITHUB_WEBHOOK_SECRET` using the `X-Hub-Signature-256` header; invalid signatures return HTTP 401 with no payload logged
  - [ ] Each valid webhook event creates a `GitHubEvent` record with: `eventType`, `payload` (raw JSON), `repoFullName`, `projectId`, `status = RECEIVED`, `receivedAt`
  - [ ] Developer can disconnect a repository from a project; this calls GitHub's `DELETE /repos/{owner}/{repo}/hooks/{hookId}` (using `Project.githubWebhookId`), nulls out `Project.githubRepoId`, `Project.githubRepoFullName`, and `Project.githubWebhookId`
  - [ ] If the stored OAuth token has been revoked, the project Settings tab shows an amber "GitHub: Reconnect required" badge and report generation for that project is blocked with a clear warning
- **UI Notes:**
  - Project Settings → GitHub tab: connection status badge; linked repo name shown with GitHub's repo avatar; "Connect GitHub" button (opens OAuth flow) or "Change Repository" + "Disconnect" buttons (if already linked)
  - Repository selector: shadcn/ui Combobox with search input; each option shows repo icon, full name, visibility badge, last-push relative date
  - Successful webhook registration shows a toast: "GitHub webhook registered for {repoFullName}"
  - Failed webhook registration (e.g. insufficient GitHub permissions) shows a destructive toast with the reason from GitHub's API error response
- **API/Logic Notes:**
  - AES-256-CBC encryption uses a random 16-byte IV per encryption operation; ciphertext is stored as `{hex_iv}:{hex_ciphertext}` in the database
  - The webhook ID for each project's repository webhook is stored exclusively on `Project.githubWebhookId`; this is the only source of truth for the per-project webhook ID used during disconnect
  - ⚠️ ASSUMPTION: GitHub OAuth App is used (not a GitHub App); OAuth tokens issued by OAuth Apps do not expire but can be revoked by the user at any time
  - GitHub retries failed webhook deliveries up to 3 times with exponential backoff; ShipDesk does not need to retry webhook receipts independently
  - If webhook registration fails due to GitHub returning 403 (insufficient permissions), store the repository link without a webhook and surface a warning: "Webhook registration failed — ShipDesk cannot automatically sync activity for this repo. Check your GitHub permissions."
- **Dependencies:** Feature 1 (Workspace), Feature 2 (Projects — a project must exist before a repo can be linked)

### Feature 4: AI Weekly Status Report Generation

- **Priority:** P0
- **User Story:** As a developer, I want ShipDesk to automatically generate a plain-English weekly project update every Friday from my GitHub activity so that my client stays informed without me writing a single word.
- **Acceptance Criteria:**
  - [ ] Every Friday at 09:00 UTC, `reportScheduler.ts` queries all `Project` records where `status = ACTIVE` and `githubRepoId IS NOT NULL`, fetches the `GitHubEvent` records for each with `receivedAt >= Monday 00:00:00.000 UTC` of the current week, and calls Gemini to generate one `Report` per project
  - [ ] Developer can manually trigger report generation at any time via the "Generate Report" button on the project detail page; this calls `POST /api/reports/generate/:projectId` and is rate-limited to 10 manual generations per project per calendar day (UTC)
  - [ ] The Gemini prompt includes a system instruction: *"You are a professional technical project manager writing a weekly status update for a non-technical business client. Be concise, positive, and jargon-free. Focus on business outcomes and visible progress, not implementation details."* Followed by structured JSON data: commits (`{ message, author, date }`), merged PRs (`{ title, description, mergedAt }`), and releases (`{ tag, name, body }`) from the past 7 days
  - [ ] Gemini response is requested as structured JSON matching `ReportContent`; if JSON parsing fails, the raw text is stored as `content.rawMarkdown` and all other structured fields (`summary`, `highlights`, `nextSteps`) are set to `null`; `content.generationWarning` is set to the parse error message to signal partial failure
  - [ ] The generated `Report` is stored with `status = DRAFT`; developer reviews the report in the UI, optionally edits it via Tiptap, and clicks "Publish to Client" to set `status = PUBLISHED` and `publishedAt = now()`
  - [ ] On publish, an email notification is sent to the client via Resend containing the report title, a 2-sentence teaser, and a "View Full Report" CTA linking to the client portal; the teaser uses `content.summary` if non-null; if `summary` is null (parse-failure path, `generationWarning` is set), the teaser falls back to the first 280 characters of `content.rawMarkdown` with a trailing "…"; if both are null, the teaser paragraph is omitted entirely from the email template
  - [ ] If a project has zero GitHub events in the reporting window, a report is still generated; the Gemini prompt explicitly states "No code activity was recorded this week" and Gemini produces a "quiet week" narrative; the report is not suppressed
  - [ ] Maximum 200 GitHub events are processed per report; if > 200 events exist in the window, the 200 most recent are used and the prompt notes "summarizing from the 200 most recent of {total} events"
  - [ ] Manual generation loading state shows three sequential animated steps (Framer Motion): "Fetching GitHub activity" → "Analyzing commits" → "Writing summary"; each step shows a spinner while running, then a green checkmark on completion
- **UI Notes:**
  - Reports tab: chronological list of report cards; each card shows: week date range, `status` badge (Draft / Published), word count, and generated-by indicator (Scheduled / Manual); clicking a card opens the full report viewer
  - Report viewer: Markdown rendered with Inter typography; "Edit" button activates Tiptap editor overlay; "Publish to Client" CTA button fixed at the bottom; "Delete Draft" button in muted text (only shown for DRAFT reports)
  - Manual generation: triggered via the "Generate Report" button in the project header; a centered modal displays the three animated steps
- **API/Logic Notes:**
  - Gemini model: `gemini-1.5-pro`; temperature: `0.4` (low variation for professional consistency); `maxOutputTokens: 2048`
  - Rate limit enforcement: `RateLimitLog` table with unique constraint on `(projectId, action, date)`; `date` is the UTC date string `YYYY-MM-DD`; `count` is incremented atomically using Prisma's `upsert` with `increment` operation; if `count >= 10`, return HTTP 429
  - Scheduled report generation is wrapped in a try/catch per project; if one project's generation fails, it logs the error and continues to the next project without stopping the batch
  - `Report.generatedBy` is set to `SCHEDULED` for Friday cron runs and `MANUAL` for developer-triggered generations
  - `Report.weekStartDate` = Monday 00:00:00.000 UTC of the ISO 8601 week containing the current UTC timestamp at the time of generation; `Report.weekEndDate` = Sunday 23:59:59.999 UTC of that same week; these are computed at generation time from `new Date()` (UTC), not from dates in the events; this rule applies to both scheduled and manual generation
  - **Multiple reports per week:** No unique constraint is placed on `(projectId, weekStartDate)`. Multiple draft reports for the same week are allowed by design — the developer may generate, discard (delete), and regenerate within the same week. The developer is responsible for ensuring only one report per week is published; the UI should display a warning toast "A draft report already exists for this week" if a draft with the same `weekStartDate` exists when the developer clicks "Generate Report", but the generation is not blocked
- **Dependencies:** Feature 2 (Projects), Feature 3 (GitHub Connection), Google Gemini API

### Feature 6: Client Invitation Flow

- **Priority:** P0
- **User Story:** As a developer, I want to invite a client to their portal via email so that they can access their project instantly without creating a password, giving them a frictionless first experience.
- **Acceptance Criteria:**
  - [ ] Developer invites a client from the project detail page by entering the client's email address and clicking "Send Invite"; this calls `POST /api/projects/:id/invite` with `{ email: string }`
  - [ ] The server: (1) finds or creates a `Client` record by email scoped to the workspace; (2) upserts a `ClientProjectAccess` record linking the client to the project (sets `grantedAt = now()`, clears any previous `revokedAt`); (3) generates a signed JWT magic link token with 7-day expiry (`{ clientId, projectId, workspaceId, type: "magic_link" }`, signed with `SESSION_SECRET`); (4) stores a SHA-256 hash of the token in `ClientInvitation.tokenHash`, invalidating any prior active token for the same client-project pair via upsert; (5) sends the invitation email via Resend
  - [ ] The magic link URL format: `https://{workspaceSlug}.portal.shipdesk.io/auth/magic?token={jwt}`
  - [ ] Client clicks the magic link; the frontend calls `POST /api/portal/auth/magic` with `{ token }` in the body; the server: (1) verifies the JWT signature using `SESSION_SECRET` first; (2) computes SHA-256 of the token and checks it matches a non-expired `ClientInvitation.tokenHash`; (3) marks `ClientInvitation.acceptedAt = now()`; (4) creates a `ClientSession` record; (5) sets a `shipdesk_client_session` cookie (`httpOnly`, `secure` in production, `sameSite: lax`, 30-day expiry matching `ClientSession.expiresAt`)
  - [ ] If the token is expired (> 7 days past `sentAt`), the server returns HTTP 401 with `{ error: "LINK_EXPIRED" }`; the frontend shows: "This link has expired. Contact your project manager to request a new invite." with no retry option
  - [ ] If the token has already been superseded (a newer token exists for the same client-project pair, meaning the SHA-256 hash does not match any active `ClientInvitation` record), the server returns HTTP 401 with `{ error: "LINK_SUPERSEDED" }`; the frontend shows a message directing the client to check their inbox for the most recent email
  - [ ] Developer can see invitation status per client on the project Settings tab: `PENDING` (sent, `acceptedAt IS NULL AND expiresAt > now()`), `ACTIVE` (`acceptedAt IS NOT NULL`), `EXPIRED` (`expiresAt < now() AND acceptedAt IS NULL`)
  - [ ] Developer can revoke a client's access at any time via `DELETE /api/projects/:projectId/clients/:clientId`; this sets `ClientProjectAccess.revokedAt = now()` and deletes all `ClientSession` records for that client **scoped to this workspace** (`WHERE clientId = :clientId AND workspaceId = workspace.id`) immediately; sessions the client holds in other workspaces are not affected
  - [ ] `clientAuth.ts` middleware checks `ClientSession.expiresAt` on every authenticated request; if `now() > session.expiresAt`, the middleware deletes the expired session record, clears the `shipdesk_client_session` cookie, and returns HTTP 401 `{ error: "SESSION_EXPIRED" }`; the frontend handles this error by redirecting to the portal root (`/{slug}`) and displaying: "Your session has expired. Use your original invitation link to sign back in."
- **UI Notes:**
  - Invite slide-over: email input with `type="email"` validation; "Send Invite" button; below the form, a list of current clients on this project with status badge and "Revoke Access" button
  - Invitation email template: workspace logo + primary color applied; subject line "You've been invited to view {Project Name}"; body includes one clear "View Your Project" CTA button; secondary text lists what the client will be able to see (reports, files, messages)
  - Client portal: first page after magic link auth shows a brief welcome message ("Welcome to your {Agency Name} client portal") before redirecting to the project overview
- **API/Logic Notes:**
  - `ClientInvitation` has a unique constraint on `(clientId, projectId)`; a new invite call performs an `upsert` — it updates the existing record's `tokenHash`, `sentAt`, and `expiresAt` rather than creating a duplicate; `acceptedAt` is reset to `null` so the status correctly shows `PENDING` again
  - Client session cookie name: `shipdesk_client_session`; value is the `ClientSession.id` UUID; the `clientAuth.ts` middleware resolves this to a `ClientSession` record, verifies the session is not expired, updates `ClientSession.lastActivityAt = now()`, and then resolves the associated `Client` record on every authenticated request
  - A client invited to multiple projects in the same workspace uses a single session cookie but their accessible project list is determined by querying all non-revoked `ClientProjectAccess` records for their `clientId`
- **Dependencies:** Feature 1 (Workspace), Feature 2 (Projects), Feature 5 (Client Portal), Resend email

### Feature 9: Scope Change Request Flow

- **Priority:** P0
- **User Story:** As a client, I want to formally submit scope change requests and as a developer I want to quote and collect payment for them so that every piece of additional work is documented, agreed upon, and paid before it begins.
- **Acceptance Criteria:**
  - [ ] Client submits a scope change request from the portal via a form: `title` (required, max 150 chars), `description` (required, max 2,000 chars, plain text), `urgency` (`LOW | MEDIUM | HIGH` radio buttons); submission creates a `ScopeChange` record with `status = PENDING` and sends a notification email to the developer
  - [ ] Developer views all pending scope changes in the project "Scope Changes" tab and in a global `/scope-changes` page; each entry shows: title, client name, submitted date, urgency badge, and current status
  - [ ] Developer submits a quote via `PATCH /api/scope-changes/:id/quote` with: `quoteDescription` (rich text HTML from Tiptap, max 10,000 chars), `quotePrice` (decimal ≥ 0.01), `quoteCurrency` (`USD | EUR | GBP | CAD | AUD`); this sets `status = QUOTED`, `quotedAt = now()`, and sends the client a notification email
  - [ ] Client views the quote in their portal and either clicks "Approve" or "Decline"; both actions call `PATCH /api/portal/scope-changes/:id/respond` with `{ decision: "APPROVED" | "DECLINED" }`; the server first verifies `ScopeChange.clientId === session.clientId` — if they do not match, return HTTP 403 `{ error: "FORBIDDEN" }` immediately (prevents one client from acting on another client's scope change); on success, sets `status = APPROVED` or `DECLINED`, `respondedAt = now()`, and triggers a developer notification email
  - [ ] The client portal scope change list (`GET /api/portal/projects/:id/scope-changes`) returns only scope changes where `clientId = session.clientId`; scope changes submitted by other clients who may also have access to the same project are never exposed
  - [ ] Immediately on `status = APPROVED`, the server calls Lemon Squeezy to create a checkout link (`POST /v1/checkouts`) with `custom_data: { scopeChangeId, type: "scope_change" }` and the quote price; the returned checkout URL is stored in `ScopeChange.paymentUrl` and the client portal immediately shows a "Pay Now" button (which opens the Lemon Squeezy checkout in a new tab)
  - [ ] Lemon Squeezy fires `order_created` to `POST /api/webhooks/lemonsqueezy`; the server verifies the signature, matches to the `ScopeChange` via `custom_data.scopeChangeId`, sets `status = PAID`, `paidAt = now()`, and `lsOrderId`; both developer and client views show a green "Paid & Added to Scope" badge
  - [ ] Developer can manually mark any scope change as `PAID` (for offline payments) via a "Mark as Paid" button guarded by a confirmation modal
- **ScopeChange Status Machine:**
  ```
  PENDING ──[developer quotes]──► QUOTED ──[client approves]──► APPROVED ──[webhook confirms]──► PAID
                                      │
                                      └──[client declines]──► DECLINED

  Any non-PAID status ──[developer manual override]──► PAID
  ```
- **UI Notes:**
  - Client portal: "Request a Change" button fixed in the project overview section; opens a centered modal form (not a slide-over, to ensure mobile usability); submission shows a success inline message (not a toast) — "Your request has been submitted. Your project manager will review it and get back to you."
  - Developer view: scope change cards with status badges — PENDING=amber, QUOTED=blue, APPROVED=indigo, DECLINED=red, PAID=green; card action area changes based on status (PENDING → "Write Quote", QUOTED → awaiting client, APPROVED → awaiting payment, DECLINED → view only, PAID → view only)
  - Quote form: slide-over from the right; Tiptap editor for the description (bold, italic, bullet list, numbered list, paragraph — no other formatting); price input with currency dropdown; "Preview" toggle shows how the client will see the quote
- **API/Logic Notes:**
  - `quoteDescription` contains HTML output from Tiptap; before storage, it is sanitized server-side using `sanitize-html` with an allowlist of: `<b>`, `<strong>`, `<i>`, `<em>`, `<u>`, `<ul>`, `<ol>`, `<li>`, `<p>`, `<br>` — all other tags and all attributes are stripped; this prevents XSS when the HTML is rendered in the client portal
  - Lemon Squeezy checkout price: `quotePrice * 100` converted to integer (cents); ensure rounding to nearest cent before multiplication (`Math.round(quotePrice * 100)`)
  - ⚠️ ASSUMPTION: Lemon Squeezy supports per-checkout price overrides on a single variant; if not, a new product/variant must be created per scope change via the API — clarify with OQ-6 before implementation
  - Lemon Squeezy webhook `X-Signature` header: HMAC-SHA256 of the raw request body using `LEMONSQUEEZY_WEBHOOK_SECRET`; must use `express.raw({ type: 'application/json' })` middleware on the webhook route to preserve the raw body before JSON parsing
- **Dependencies:** Feature 2 (Projects), Feature 5 (Client Portal), Feature 6 (Client Invitation), Lemon Squeezy

## 10. UI/UX REQUIREMENTS

### 10.1 Design Principles

1. **Desktop-first, fully responsive:** Primary design target is 1280px desktop; all layouts must be fully usable at 375px (iPhone SE) without horizontal scrolling or broken touch targets
2. **WCAG 2.1 AA compliance:** All interactive elements have a 2px visible focus ring; color contrast ratio ≥ 4.5:1 for body text, ≥ 3:1 for large text and UI components; all non-decorative images have descriptive `alt` text
3. **Professional minimal aesthetic:** No decorative illustrations in the app shell (landing page excepted); whitespace is used generously; every UI element must have a clear purpose; gradients are reserved for the landing page only
4. **Dark mode supported:** All components implement Tailwind's `dark:` variants; theme is toggled via the `class` strategy on `<html>`; user preference is saved to `localStorage` and applied before first paint to prevent flash
5. **Skeleton-first loading states:** Every data-fetching component renders a content-matching skeleton loader (not a spinner) while the TanStack Query request is in `loading` state; the skeleton matches the proportions of the loaded content to prevent layout shift
6. **Explicit error states:** No blank screens on error; every error state shows an icon, a plain-English description of what went wrong, and a concrete retry action (button or link)

## 11. AUTHENTICATION & AUTHORIZATION

### Auth Methods

| Party | Method | Token Location | Session Lifetime |
|-------|--------|---------------|-----------------|
| Developer | Clerk JWT (email/password or Google OAuth) | `Authorization: Bearer {token}` header | Managed by Clerk (sliding, ~1 hour token with refresh) |
| Client | Session cookie referencing `ClientSession` DB record | `shipdesk_client_session` cookie (`httpOnly`, `secure`, `sameSite: lax`) | 30 days from creation; `lastActivityAt` updated on each request but no sliding window in v1.0 |
| Webhooks (GitHub, Lemon Squeezy) | HMAC-SHA256 signature on raw request body | `X-Hub-Signature-256` / `X-Signature` header | Per-request; no persistent session; failures return HTTP 401 |
| Clerk webhooks | svix signature | `svix-id`, `svix-timestamp`, `svix-signature` headers | Per-request; failures return HTTP 401 |

### Role Permissions

| Role | Access Scope | Permitted Actions |
|------|-------------|-------------------|
| Developer (Workspace Owner) | All resources within their workspace | Full CRUD on all workspace resources; GitHub OAuth; workspace branding; Lemon Squeezy checkout creation; publish reports |
| Client | Projects they have been explicitly invited to (non-revoked `ClientProjectAccess`) | Read published reports; read/upload/delete own files; read/send messages; submit scope change requests; approve/decline quotes; view and pay invoices |
| Public (unauthenticated) | Public endpoints only | Fetch workspace branding; submit magic link token; receive webhooks; health check |

### Authorization Enforcement Rules

1. Every developer API handler resolves the Clerk `auth.userId` to `User.clerkId`, then to `User.id`, then verifies `resource.workspaceId === workspace.id` where `workspace.ownerId === User.id`; any mismatch returns HTTP 403
2. Every client portal API handler resolves `req.cookies.shipdesk_client_session` to a `ClientSession` record and then verifies `ClientProjectAccess` exists where `clientId = session.clientId AND projectId = :id AND revokedAt IS NULL`; any failure returns HTTP 403
3. Client JWT tokens are explicitly rejected on developer routes; developer Clerk JWTs are explicitly rejected on portal routes
4. File deletion: developers may delete any `ProjectFile` in their workspace; clients may only delete files where `uploadedBy === session.clientId AND uploaderType === 'CLIENT'`; violations return HTTP 403
5. Report access on the client portal: only reports with `status = PUBLISHED` are returned; `DRAFT` reports are never exposed to clients — the endpoint returns HTTP 404 rather than HTTP 403 to avoid leaking draft existence
6. The scope-change respond endpoint (`PATCH /api/portal/scope-changes/:id/respond`) additionally verifies `ScopeChange.clientId === session.clientId` before any state transition; a mismatch returns HTTP 403 `{ error: "FORBIDDEN" }` to prevent one client from approving or declining another client's scope change request

## 12. ERROR HANDLING & EDGE CASES

- [ ] **Empty or invalid form submission:** All request bodies are validated server-side using `zod` schemas; failed validation returns HTTP 400 with `{ error: "VALIDATION_ERROR", fields: Record<string, string[]> }`; the frontend maps `fields` to inline field-level error messages under the relevant input
- [ ] **Workspace already exists on `POST /api/workspace`:** If the authenticated developer already has a workspace (enforced by unique constraint on `Workspace.ownerId`), returns HTTP 409 `{ error: "WORKSPACE_ALREADY_EXISTS" }`; the onboarding redirect guard prevents this in normal flow, but the server enforces it as a hard constraint; frontend should not reach this state if `GET /api/workspace` is called before rendering `/onboarding`
- [ ] **Client session expired (30-day TTL):** `clientAuth.ts` detects `now() > session.expiresAt`; deletes the expired `ClientSession` record; clears the `shipdesk_client_session` cookie; returns HTTP 401 `{ error: "SESSION_EXPIRED" }`; frontend intercepts this at the Axios response level and redirects to the portal root with message: "Your session has expired. Use your original invitation link to sign back in." with no retry button
- [ ] **GitHub API unavailable during report generation:** `githubService.ts` wraps all calls in a try/catch; on network error or HTTP 5xx, the scheduler logs the error, skips the report for that project, and continues to the next project; the developer sees a "GitHub data unavailable — report was not generated. Click to retry." warning on the project's Reports tab
- [ ] **Gemini API unavailable or quota exceeded (HTTP 429 / 5xx):** The report generation service catches the error, does NOT persist any partial `Report` record, and logs the failure; the developer sees "Report generation failed. Gemini service is unavailable. Click to retry."; retry is subject to the same rate limit
- [ ] **Gemini JSON parsing failure:** A `Report` record IS persisted with `content.rawMarkdown` containing the raw text response and `content.generationWarning` containing the parse error message; all other `ReportContent` fields are null; the developer sees a yellow warning banner on the report: "AI could not structure this report — raw output is shown below."
- [ ] **Unauthorized resource access:** Any `workspaceId` mismatch or missing `ClientProjectAccess` returns HTTP 403 `{ error: "FORBIDDEN" }`; the response does not reveal whether the resource exists (no difference between "not found" and "not yours" from the API's perspective)
- [ ] **File upload exceeds 50 MB:** Rejected client-side before the upload begins with an inline error: "File too large — maximum 50 MB per file"; Cloudinary upload preset enforces `max_file_size: 52428800` bytes as a server-side backup
- [ ] **Magic link expired (> 7 days):** Returns HTTP 401 `{ error: "LINK_EXPIRED" }`; frontend renders: "This link has expired. Contact your project manager to request a new invitation." with no retry button
- [ ] **Magic link superseded (a newer token exists):** Returns HTTP 401 `{ error: "LINK_SUPERSEDED" }`; frontend renders: "This link is no longer valid. Check your email for a more recent invitation."
- [ ] **GitHub webhook signature invalid:** Returns HTTP 401 immediately; the raw payload is not parsed or stored; the event is logged with source IP for security monitoring
- [ ] **Clerk webhook signature invalid:** Returns HTTP 401 immediately; no database operations are performed
- [ ] **Lemon Squeezy webhook duplicate delivery:** Before processing, the handler checks whether `lsOrderId` already exists in the `Invoice` or `ScopeChange` table; if found, returns HTTP 200 with `{ received: true, skipped: "duplicate" }` without reprocessing
- [ ] **Project at 50-active-project limit:** `POST /api/projects` returns HTTP 422 `{ error: "ACTIVE_PROJECT_LIMIT_REACHED", limit: 50 }` with user-facing message: "You've reached the limit of 50 active projects. Archive some projects to create new ones."
- [ ] **Project at 500-file limit:** `POST /api/projects/:id/files` returns HTTP 422 `{ error: "FILE_LIMIT_REACHED", limit: 500 }`
- [ ] **Client accesses portal after access is revoked:** `clientAuth.ts` middleware checks `ClientProjectAccess.revokedAt`; if revoked, clears the session cookie and returns HTTP 401 `{ error: "ACCESS_REVOKED" }`; frontend redirects to portal root with message: "Your access to this project has been removed. Contact your project manager."
- [ ] **Manual report generation rate limit exceeded:** Returns HTTP 429 `{ error: "RATE_LIMIT_EXCEEDED", limit: 10, resetAt: "<ISO timestamp of midnight UTC today>" }`; the `resetAt` value is dynamically computed as the start of the next UTC calendar day
- [ ] **Cloudinary upload failure mid-stream:** The `ProjectFile` DB record is only created AFTER Cloudinary confirms a successful upload; if Cloudinary returns an error, the frontend shows: "Upload failed — please try again"; no orphaned DB records are created
- [ ] **Scope change approved but Lemon Squeezy checkout creation fails:** The `status = APPROVED` is still persisted; a `paymentUrl` generation error is logged; the developer sees a warning: "Quote approved but payment link could not be created. Click here to retry." with a retry button that re-calls the Lemon Squeezy checkout endpoint
- [ ] **Attempt to delete a PAID invoice:** Returns HTTP 409 `{ error: "CANNOT_DELETE_PAID_INVOICE" }`; frontend shows a toast: "Paid invoices cannot be deleted. Archive the project if it is no longer active."
- [ ] **Workspace slug conflict on initial creation:** Returns HTTP 409 `{ error: "SLUG_TAKEN" }`; this should not occur if the client-side availability check works correctly, but the server enforces uniqueness as a final guard via a database unique constraint
- [ ] **Invalid `Project.status` transition attempt:** Returns HTTP 422 `{ error: "CANNOT_REACTIVATE_COMPLETED_PROJECT" }` with a user-facing message: "Completed projects cannot be reactivated. Create a new project to continue this work."
- [ ] **Message send rate limit exceeded:** Returns HTTP 429 `{ error: "RATE_LIMIT_EXCEEDED" }`; frontend shows an inline message below the input: "You're sending messages too quickly. Wait a moment and try again."

## 13. PERFORMANCE & SECURITY REQUIREMENTS

### Performance

- [ ] **First Contentful Paint (FCP):** ≤ 1.5 seconds on a simulated 4G connection (40 Mbps, 40ms RTT) in Lighthouse; enforced by code splitting at the route level using Vite's dynamic `import()` and TanStack Query prefetching of workspace data on auth
- [ ] **Time to Interactive (TTI):** ≤ 2.5 seconds on the same simulated 4G connection
- [ ] **API response time (non-AI endpoints):** p95 ≤ 300ms under normal load (< 100 concurrent requests); measured via Railway's built-in metrics
- [ ] **AI report generation:** Treated as an async task; the user receives an animated loading state while waiting; there is no timeout on the frontend — the request remains open; the backend enforces a 60-second server-side timeout on the Gemini API call, after which it returns a generation failure without persisting a partial record

### Security

- [ ] **Input sanitization:** All string inputs validated by `zod` on the server (type, length, format); fields that accept HTML (`quoteDescription`) are sanitized with `sanitize-html` before storage using a strict allowlist (`<b>`, `<strong>`, `<i>`, `<em>`, `<u>`, `<ul>`, `<ol>`, `<li>`, `<p>`, `<br>` only); plain-text fields have all HTML stripped (empty allowlist)
- [ ] **No passwords stored:** Clerk manages all developer authentication credentials; the ShipDesk database contains no password fields
- [ ] **API rate limiting:** `express-rate-limit` applied: 60 req/min per IP on the branding endpoint; 10 req/min per IP on the magic link endpoint; 100 req/min per authenticated developer on all other routes; 30 req/min per IP on the subdomain check endpoint; 30 messages/min per sender on message endpoints
- [ ] **Secrets never exposed to client:** `GEMINI_API_KEY`, `CLOUDINARY_API_SECRET`, `CLERK_SECRET_KEY`, `LEMONSQUEEZY_API_KEY`, `RESEND_API_KEY`, `GITHUB_CLIENT_SECRET`, `AES_ENCRYPTION_KEY`, `SESSION_SECRET`, and `DATABASE_URL` must only appear in server-side `process.env`; the Vite build exposes only `VITE_CLERK_PUBLISHABLE_KEY` and `VITE_API_BASE_URL` (both prefixed with `VITE_`) to the client bundle
- [ ] **HTTPS enforced:** Both Railway and Vercel enforce HTTPS at the platform level; Express sets `Strict-Transport-Security: max-age=31536000; includeSubDomains` in production
- [ ] **CORS policy:** Express `cors()` configured with `origin: [process.env.FRONTEND_URL, /^https:\/\/[a-z0-9-]+\.portal\.shipdesk\.io$/]` in production; `http://localhost:5173` allowed only when `NODE_ENV === 'development'`; `credentials: true` required for client session cookies
- [ ] **Prisma binary targets for Railway:** `schema.prisma` declares `binaryTargets = ["native", "linux-musl-openssl-3.0.x", "linux-musl-arm64-openssl-3.0.x"]` to cover both x86 and ARM Railway deployments
- [ ] **SQL injection prevention:** All database access goes through Prisma's parameterized query builder; raw SQL (`$queryRaw`) is avoided; if required for a specific query, all parameters must be passed as Prisma template literal placeholders (not string concatenation)
- [ ] **Webhook raw body preservation:** The webhook routes (`/api/webhooks/*`) must use `express.raw({ type: 'application/json' })` middleware BEFORE `express.json()` to preserve the raw request body for HMAC signature verification; using the parsed body for signature checks will always fail
- [ ] **Session security:** `shipdesk_client_session` cookie is `httpOnly: true`, `secure: true` (production), `sameSite: 'lax'`; session ID is a UUID v4 (128-bit entropy); sessions are invalidated server-side on logout or access revocation, making cookie theft immediately ineffective
- [ ] **GitHub token encryption:** `GitHubConnection.accessTokenEncrypted` stores tokens as AES-256-CBC ciphertext with a random 16-byte IV per encryption, formatted as `{hex_iv}:{hex_ciphertext}`; the plaintext token is never written to logs, errors, or any storage other than this field; same format applies to `LinearConnection` and `VercelConnection` encrypted fields
- [ ] **Cloudinary upload authorization:** All file uploads use server-generated signed upload parameters that expire 60 seconds after generation; unsigned upload presets are never used; the server validates that the requesting user has access to the target `projectId` before issuing any signature
- [ ] **Key separation:** `SESSION_SECRET` (≥ 32 chars, any random string) is used exclusively for JWT and cookie signing; `AES_ENCRYPTION_KEY` (exactly 64 hex chars representing 256 bits) is used exclusively for AES-256 encryption of third-party tokens; these keys serve different cryptographic purposes and must never be set to the same value

## 18. GLOSSARY

| Term | Definition |
|------|-----------|
| **Workspace** | A developer's top-level account container in ShipDesk. Contains all projects, clients, branding settings, and the subdomain slug. One workspace per developer in v1.0. |
| **Slug** | The URL-safe identifier used as the subdomain for a workspace's client portal (e.g. the slug `acme-dev` creates the portal at `acme-dev.portal.shipdesk.io`). Stored in lowercase. Immutable after creation. |
| **Client portal** | The white-label, branded web interface where a client views their project's reports, files, invoices, and messages. Served at `{slug}.portal.shipdesk.io`. Requires no password — accessed via magic link. |
| **Magic link** | A signed JWT embedded in an email URL that authenticates a client to their portal without requiring a password. Valid for 7 days; invalidated on re-invite. The JWT is signed with `SESSION_SECRET`; only its SHA-256 hash is stored in the database. |
| **GitHubEvent** | A database record of a raw GitHub webhook payload (push, pull request, or release) associated with a specific project. These records are the source data for AI report generation. |
| **Report** | An AI-generated, developer-editable plain-English summary of a project's GitHub activity for a given 7-day window. Exists in `DRAFT` (visible to developer only) or `PUBLISHED` (visible to client) states. |
| **ReportContent** | The structured JSON content of a report, containing `summary`, `highlights`, `nextSteps`, `rawMarkdown`, and `generationWarning` fields parsed from the Gemini API response. Stored as a JSON column in the `Report` table. |
| **ReportMeta** | A lightweight subset of `Report` fields returned by list endpoints (excludes `content`). Contains: `id`, `projectId`, `weekStartDate`, `weekEndDate`, `title`, `status`, `generatedAt`, `publishedAt`, `generatedBy`. |
| **ScopeChange** | A formal change request submitted by a client, reviewed and quoted by the developer, and optionally paid via Lemon Squeezy. Follows a defined state machine: `PENDING → QUOTED → APPROVED/DECLINED → PAID`. |
| **Invoice** | A milestone payment record created by the developer, linked to a Lemon Squeezy checkout URL. Status is automatically updated to `PAID` via webhook on successful client payment. |
| **ClientProjectAccess** | A join table record that grants a specific client access to a specific project. Revocable at any time by setting `revokedAt`. A client may have access to multiple projects within the same workspace via separate records. |
| **ClientSession** | A server-side session record created when a client authenticates via magic link. Referenced by the `shipdesk_client_session` cookie. Enables access revocation by deleting this record server-side. |
| **Lemon Squeezy** | The Merchant of Record payment platform used for all monetary transactions in ShipDesk (invoices and scope change payments). Handles global VAT/GST compliance automatically. |
| **Gemini** | Google's multimodal AI model (`gemini-1.5-pro`) used to generate plain-English weekly project status reports from structured GitHub activity data. |
| **Tiptap** | A headless, extensible rich text editor used in ShipDesk for the scope change quote description and for developer editing of AI-generated reports before publishing. |
| **Framer Motion** | A React animation library used for report generation loading state animations, onboarding checklist entrance animations, and page transitions. |
| **Cron job** | A scheduled background task managed by `node-cron`. ShipDesk has two: (1) `0 9 * * 5` — Fridays at 09:00 UTC, generate weekly reports; (2) `5 0 * * *` — daily at 00:05 UTC, mark overdue invoices. |
| **AES-256-CBC** | A symmetric encryption algorithm used to encrypt GitHub, Linear, and Vercel OAuth tokens at rest. Uses a random 16-byte IV per encryption, stored alongside the ciphertext as `{hex_iv}:{hex_ciphertext}`. |
| **HMAC-SHA256** | A message authentication code algorithm used to verify the integrity and authenticity of incoming GitHub and Lemon Squeezy webhook payloads. |
| **Cursor-based pagination** | A pagination strategy where the client provides a `before` cursor (the `createdAt` ISO 8601 timestamp of the oldest loaded item) to fetch the previous page; more reliable than offset-based pagination for real-time data like message threads. |
| **White-label** | A customization model where ShipDesk's client portal displays the developer's or agency's branding (logo, colors, name) with no visible ShipDesk attribution. |
| **P0 / P1 / P2** | Priority designations: P0 = must-have for v1.0 MVP launch; P1 = important, targeted for v2; P2 = nice-to-have, planned for v3+. |
| **MoR (Merchant of Record)** | A company that takes legal and financial responsibility for processing payments, including VAT/GST compliance. Lemon Squeezy acts as MoR for all ShipDesk transactions. |
| **Signed upload parameters** | A set of server-generated Cloudinary parameters (signature, timestamp, api_key, upload_preset, folder) that authorize a direct browser-to-Cloudinary file upload for a fixed 60-second window. Replaces unsigned upload presets for security. |
| **LinearConnection** | A database record storing a developer's encrypted Linear OAuth credentials (access token + refresh token) and token expiry metadata, enabling automatic Linear issue fetching during report generation. |
| **VercelConnection** | A database record storing a developer's encrypted Vercel API token scoped to a specific ShipDesk project, enabling deployment event fetching during report generation. |
